## Инструкция по развертыванию и настройке PostgreSQL на сервере Ubuntu 20.04

### 1. Установка PostgreSQL

Установка выполняется с помощью пакетного менеджера `apt`.

```bash
sudo apt update
sudo apt install postgresql -y
```

**Объяснение команд и ключей:**

| Команда / Ключ           | Назначение                                                                                                                                                                                                                                     |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `sudo`                   | Выполнение команды с привилегиями суперпользователя, необходимыми для установки пакетов.                                                                                                                                                       |
| `apt update`             | Обновление локального индекса доступных пакетов из подключенных репозиториев.                                                                                                                                                                  |
| `apt install postgresql` | Установка метапакета `postgresql`, который тянет актуальную поддерживаемую версию СУБД (в репозитории Ubuntu 20.04 по умолчанию — 12, но при подключении стороннего репозитория или обновлении дистрибутива может быть установлена версия 16). |
| `-y`                     | Автоматическое подтверждение запроса на установку, исключает интерактивное взаимодействие.                                                                                                                                                     |

После установки служба PostgreSQL запускается автоматически и добавляется в автозагрузку.

### 2. Проверка состояния службы

```bash
sudo systemctl status postgresql
```

**Назначение:** Убедиться, что служба активна (`Active: active (exited)` или `active (running)` в зависимости от версии systemd). В случае ошибок будет выведена диагностическая информация.

Управление службой:

| Команда                             | Действие                                                            |
| ----------------------------------- | ------------------------------------------------------------------- |
| `sudo systemctl start postgresql`   | Запуск службы                                                       |
| `sudo systemctl stop postgresql`    | Остановка службы                                                    |
| `sudo systemctl restart postgresql` | Перезапуск службы                                                   |
| `sudo systemctl enable postgresql`  | Добавление в автозагрузку (выполняется автоматически при установке) |

### 3. Локальный доступ через консольный клиент `psql`

PostgreSQL использует модель аутентификации `peer` для локальных подключений системного пользователя `postgres`. Это означает, что при входе в ОС под именем `postgres` доступ к СУБД предоставляется без пароля.

**Переключение на пользователя `postgres`:**

```bash
sudo -i -u postgres
```

| Ключ | Назначение |
|------|------------|
| `-i` | Запуск оболочки входа (login shell), что загружает окружение пользователя `postgres`. |
| `-u postgres` | Указание целевого пользователя. |

После выполнения приглашение изменится на `postgres@hostname:~$`.

**Запуск интерактивной консоли PostgreSQL:**

```bash
psql
```

При успешном подключении отобразится приглашение `postgres=#`.

**Альтернативный однострочный вариант (без переключения пользователя):**

```bash
sudo -u postgres psql
```

### 4. Создание базы данных и пользователя (роли)

Все действия выполняются в SQL-консоли `psql` от имени суперпользователя `postgres`.

#### 4.1. Создание пользователя (роли) с паролем

```sql
CREATE USER testuser WITH ENCRYPTED PASSWORD 'testpass';
```

| Элемент синтаксиса | Назначение |
|--------------------|------------|
| `CREATE USER` | Синоним `CREATE ROLE` с неявным добавлением атрибута `LOGIN`. Создает учетную запись, способную подключаться к СУБД. |
| `testuser` | Имя роли. **Заменить на фактическое имя пользователя при необходимости.** |
| `WITH ENCRYPTED PASSWORD` | Указание пароля, который будет храниться в зашифрованном виде с использованием алгоритма SCRAM-SHA-256 (по умолчанию в PostgreSQL 10+). |
| `'testpass'` | Пароль в открытом виде. **Заменить на стойкий пароль в промышленной среде.** |

#### 4.2. Создание базы данных с указанием владельца

```sql
CREATE DATABASE testdb OWNER testuser;
```

| Элемент | Назначение |
|---------|------------|
| `CREATE DATABASE` | Создание новой базы данных. |
| `testdb` | Имя базы данных. **Заменить на требуемое имя при необходимости.** |
| `OWNER testuser` | Назначение владельца базы данных. Владелец имеет все привилегии внутри БД, включая создание схем, таблиц и управление правами. |

#### 4.3. Предоставление полных привилегий пользователю на базу данных (опционально)

```sql
GRANT ALL PRIVILEGES ON DATABASE testdb TO testuser;
```

| Элемент | Назначение |
|---------|------------|
| `GRANT ALL PRIVILEGES` | Предоставление всех прав на указанный объект (в данном случае — базу данных). |
| `ON DATABASE testdb` | Указание объекта, на который распространяются права. |
| `TO testuser` | Получатель прав. |

**Примечание:** При создании базы данных с явным указанием `OWNER` команда `GRANT ALL PRIVILEGES` избыточна, так как владелец автоматически получает полный контроль. Команда приведена для случаев, когда БД создана без указания владельца или права нужно выдать дополнительной роли.

**Выход из `psql`:**

```sql
\q
```

**Выход из сессии пользователя `postgres` (если использовалось `sudo -i`):**

```bash
exit
```

### 5. Настройка PostgreSQL для приема удаленных подключений

По умолчанию PostgreSQL слушает только локальный интерфейс `127.0.0.1`. Для доступа из внешней сети (включая Windows-клиент с pgAdmin) необходимо изменить два конфигурационных файла.

#### 5.1. Изменение `postgresql.conf`

Файл расположен в директории конфигурации PostgreSQL. Узнать путь можно командой:

```bash
sudo -u postgres psql -c "SHOW config_file;"
```

Обычно для Ubuntu это `/etc/postgresql/16/main/postgresql.conf` (версия может отличаться).

Откройте файл в текстовом редакторе:

```bash
sudo nano /etc/postgresql/16/main/postgresql.conf
```

Найдите параметр `listen_addresses` и измените его значение.

**Варианты настройки:**

| Значение | Результат |
|----------|-----------|
| `listen_addresses = 'localhost'` | Принимать соединения только с локальной петли (127.0.0.1). Значение по умолчанию. |
| `listen_addresses = '*'` | Слушать все доступные сетевые интерфейсы (включая внешний IP-адрес). Рекомендуется для простоты настройки при наличии межсетевого экрана. |
| `listen_addresses = '192.168.1.100,10.0.0.5'` | Явное перечисление IP-адресов интерфейсов, на которых PostgreSQL должен принимать подключения. |

**Рекомендуемое значение для данной инструкции:** `'*'`.

**Изменение строки в редакторе nano:**
- Найти строку `#listen_addresses = 'localhost'` (символ `#` означает комментарий).
- Раскомментировать (удалить `#`) и заменить значение:

```
listen_addresses = '*'
```

- Сохранить изменения: `Ctrl+O`, `Enter`. Выйти из редактора: `Ctrl+X`.

**Обоснование:** Без изменения этого параметра СУБД не будет принимать TCP-соединения с внешних IP-адресов, даже если правила файла `pg_hba.conf` разрешают подключение.

#### 5.2. Изменение `pg_hba.conf`

Файл управления аутентификацией на основе хостов (`HBA` — Host-Based Authentication). Расположен в той же директории, что и `postgresql.conf`.

```bash
sudo nano /etc/postgresql/16/main/pg_hba.conf
```

Добавьте строку, разрешающую подключение к базе `testdb` для пользователя `testuser` с любого IP-адреса (или с конкретного диапазона) с использованием MD5-пароля.

**Синтаксис записи:**

```
# TYPE  DATABASE        USER            ADDRESS                 METHOD
host    testdb          testuser        all                     scram-sha-256
```

**Расшифровка полей:**

| Поле | Значение | Назначение |
|------|----------|------------|
| `TYPE` | `host` | Тип соединения — TCP/IP с аутентификацией (не SSL и не GSSAPI). Альтернативы: `local` (Unix-сокет), `hostssl` (только SSL), `hostnossl` (только без SSL). |
| `DATABASE` | `testdb` | Имя базы данных, для которой действует правило. **Заменить на фактическое имя БД.** Можно указать `all` для всех баз. |
| `USER` | `testuser` | Имя роли. **Заменить на фактическое имя пользователя.** Можно указать `all` для всех ролей. |
| `ADDRESS` | `all` | Сетевой адрес клиента. `all` соответствует `0.0.0.0/0` (любой IPv4-адрес) и `::/0` (любой IPv6-адрес). Для ограничения доступа можно указать конкретную подсеть, например, `192.168.1.0/24`. |
| `METHOD` | `scram-sha-256` | Метод аутентификации. `scram-sha-256` — современный безопасный метод с хешированием пароля. Альтернативы: `md5` (менее безопасен), `password` (пароль открытым текстом, не рекомендуется), `trust` (без пароля, **категорически не рекомендуется для внешних сетей**). |

**Важные замечания по расположению записи:**
- Записи обрабатываются последовательно сверху вниз.
- Если выше есть более общая запись (например, `host all all 0.0.0.0/0 md5`), она может перекрыть добавленную. Рекомендуется размещать специфичные правила выше общих.
- После изменения файла требуется перезапуск службы PostgreSQL.

#### 5.3. Перезапуск службы PostgreSQL

```bash
sudo systemctl restart postgresql
```

### 6. Настройка межсетевого экрана (UFW)

По умолчанию PostgreSQL использует порт `5432`. Если в системе активен UFW (Uncomplicated Firewall), необходимо разрешить входящие соединения на этот порт.

**Проверка статуса UFW:**

```bash
sudo ufw status
```

Если вывод содержит `Status: active`, необходимо добавить правило.

**Добавление разрешающего правила:**

```bash
sudo ufw allow 5432/tcp
```

| Команда / Аргумент | Назначение |
|--------------------|------------|
| `ufw allow` | Добавление разрешающего правила. |
| `5432` | Номер порта PostgreSQL по умолчанию. |
| `/tcp` | Протокол транспортного уровня. PostgreSQL использует TCP. |

**Ограничение доступа по IP-адресу (рекомендуется для промышленной эксплуатации):**

```bash
sudo ufw allow from 192.168.1.100 to any port 5432 proto tcp
```

Заменить `192.168.1.100` на IP-адрес Windows-машины с pgAdmin.

**Если UFW не активен**, правило будет добавлено, но не вступит в силу до включения брандмауэра. Включение UFW выполняется командой `sudo ufw enable`. **Внимание:** перед включением убедитесь, что разрешен доступ по SSH (порт 22), иначе возможна потеря соединения.

### 7. Проверка локального подключения с использованием созданной учетной записи

После настройки удаленного доступа следует убедиться, что локальное подключение к БД под созданной ролью работает корректно.

```bash
psql -h localhost -U testuser -d testdb -W
```

| Ключ | Назначение |
|------|------------|
| `-h localhost` | Указывает хост для подключения. `localhost` означает подключение через TCP/IP на интерфейс 127.0.0.1. |
| `-U testuser` | Имя пользователя (роли) для аутентификации. **Заменить на фактическое.** |
| `-d testdb` | Имя базы данных для подключения. **Заменить на фактическое.** |
| `-W` | Принудительный запрос пароля. Даже если пароль не задан или используется peer-аутентификация, клиент запросит пароль. |

После ввода пароля `testpass` должно отобразиться приглашение `testdb=>`.

**Альтернативный вариант без указания пароля в командной строке (пароль запрашивается интерактивно):**

```bash
psql -h localhost -U testuser -d testdb
```

Для выхода выполнить `\q`.

### 8. Подключение из внешнего клиента pgAdmin (Windows)

После успешной настройки серверной части можно подключаться с помощью графического инструмента pgAdmin 4.

**Параметры подключения в pgAdmin:**

1.  Откройте pgAdmin.
2.  В дереве объектов нажмите правой кнопкой мыши на **Servers** → **Create** → **Server...**.
3.  На вкладке **General** в поле **Name** укажите произвольное имя подключения (например, `Test Server`).
4.  Перейдите на вкладку **Connection** и заполните поля:

| Поле | Значение | Примечание |
|------|----------|------------|
| **Host name/address** | IP-адрес или доменное имя сервера Ubuntu | Например, `192.168.1.10` или `server.example.com`. |
| **Port** | `5432` | Порт по умолчанию. Если изменялся в `postgresql.conf`, указать актуальный номер. |
| **Maintenance database** | `testdb` | Имя созданной базы данных. |
| **Username** | `testuser` | Имя созданной роли. |
| **Password** | `testpass` | Пароль, заданный при создании роли. |
| **Save password?** | Опционально | Включить для сохранения пароля в зашифрованном хранилище pgAdmin. |

5.  Нажмите **Save**.

При корректной настройке соединение будет установлено, и в дереве объектов появится список схем, таблиц и других объектов базы данных `testdb`.

**Диагностика проблем подключения:**

- **Тайм-аут соединения:** Проверить доступность порта 5432 с Windows-машины (команда `telnet <IP-сервера> 5432` или `Test-NetConnection <IP-сервера> -Port 5432` в PowerShell). При недоступности — проблема в сетевом экране (локальный брандмауэр Windows, корпоративный межсетевой экран) или UFW на сервере.
- **Ошибка аутентификации:** Проверить правильность имени пользователя и пароля, убедиться, что метод аутентификации в `pg_hba.conf` совместим с предоставляемым паролем (например, `scram-sha-256` требует, чтобы пароль хранился в соответствующем хеше).
- **Отсутствие записи в `pg_hba.conf`:** Проверить, что IP-адрес клиента попадает под правило `ADDRESS` (если указана конкретная подсеть).

### 9. Удаление тестовой базы данных и пользователя

Для удаления тестовых объектов необходимо подключиться к PostgreSQL с правами суперпользователя (например, `postgres`), так как удаляемая база данных может иметь активные подключения, а роль — принадлежащие ей объекты.

#### 9.1. Подключение под суперпользователем

```bash
sudo -u postgres psql
```

#### 9.2. Завершение активных подключений к базе данных (при необходимости)

Перед удалением базы данных все подключения к ней должны быть завершены.

```sql
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE datname = 'testdb' AND pid <> pg_backend_pid();
```

| Элемент | Назначение |
|---------|------------|
| `pg_terminate_backend(pid)` | Функция, принудительно разрывающая соединение с указанным идентификатором процесса (PID). |
| `pg_stat_activity` | Системное представление, содержащее информацию о текущих подключениях. |
| `WHERE datname = 'testdb'` | Фильтр по имени удаляемой базы данных. |
| `AND pid <> pg_backend_pid()` | Исключение текущего сеанса, чтобы не разорвать собственное подключение. |

#### 9.3. Удаление базы данных

```sql
DROP DATABASE IF EXISTS testdb;
```

| Ключевое слово | Назначение |
|----------------|------------|
| `DROP DATABASE` | Удаление базы данных. |
| `IF EXISTS` | Предотвращает ошибку в случае, если база данных уже не существует. |

#### 9.4. Удаление пользователя (роли)

```sql
DROP USER IF EXISTS testuser;
```

| Ключевое слово | Назначение |
|----------------|------------|
| `DROP USER` | Удаление роли с атрибутом LOGIN. |
| `IF EXISTS` | Предотвращает ошибку, если роль отсутствует. |

**Важно:** Если роль владеет какими-либо объектами в других базах данных, команда `DROP USER` завершится ошибкой. В таком случае необходимо либо передать владение (`REASSIGN OWNED BY`), либо удалить принадлежащие объекты (`DROP OWNED BY`).

**Пример корректного удаления роли с передачей прав:**

```sql
REASSIGN OWNED BY testuser TO postgres;
DROP OWNED BY testuser;
DROP USER testuser;
```

Для тестового сценария, когда роль владеет только удаляемой базой данных, эти шаги не требуются.

#### 9.5. Выход из psql

```sql
\q
```

### Заключение

Инструкция описывает полный цикл развертывания PostgreSQL на Ubuntu 20.04, настройку локального и удаленного доступа, создание тестовой базы данных и пользователя, а также процедуру удаления тестовых объектов. При использовании в промышленной среде необходимо заменить значения `testdb`, `testuser`, `testpass` на соответствующие корпоративным стандартам и принять меры по ограничению доступа (конкретные IP-адреса в `pg_hba.conf` и UFW, использование SSL-соединений).
