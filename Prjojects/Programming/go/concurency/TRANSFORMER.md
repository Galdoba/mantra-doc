**Область применения**: по сути декоратор. 
- Принимает канал
- ловит сообщение
- Меняет сообщение
- возвращает такой же канал с измененными значениями

```go

package main

import (
	"fmt"
)

func Transform[T any](inputChs <-chan T, transformFunc func(T) T) <-chan T {
	outputCh := make(chan T)

	go func() {
		defer close(outputCh)
		for number := range inputChs {
			outputCh <- transformFunc(number)
		}
	}()

	return outputCh
}

func main() {
	channel := make(chan int)

	go func() {
		defer close(channel)

		for i := 0; i < 100; i += 3 {
			channel <- i
		}
	}()
	multiplicate := func(value int) int {
		return value * value
	}

	for number := range Transform(channel, multiplicate) {
		fmt.Println(number)
	}

}
```


## Объяснение простыми словами

Этот код создаёт «конвейер» для обработки данных, которые текут через каналы (каналы в Go — это как трубы, по которым можно передавать значения между горутинами).

**Суть:**  
Есть функция `Transform`. Она берёт канал с данными любого типа (`T`), применяет к каждому значению функцию-преобразователь и возвращает **новый** канал с уже преобразованными значениями. При этом исходный канал остаётся неизменным.

В примере `main`:
1. Запускается горутина (лёгкий поток), которая посылает в канал `channel` числа: 0, 3, 6, …, 99.
2. Функция `multiplicate` возводит число в квадрат.
3. Вызов `Transform(channel, multiplicate)` создаёт новый канал, куда автоматически попадают квадраты чисел.
4. Цикл `range` читает из этого нового канала и печатает результаты.

**Что получается на выходе?**  
0, 9, 36, 81, …, 9801 (квадраты чисел, кратных трём).

## Что делает `Transform` шаг за шагом

1. Создаёт **новый** канал `outputCh`.
2. Запускает горутину, которая:
   - читает из **входного** канала `inputChs` (пока он не закроется);
   - для каждого прочитанного значения `number` вызывает `transformFunc(number)`;
   - отправляет результат в `outputCh`;
   - когда входной канал закрыт и все данные прочитаны, **закрывает** `outputCh`.
3. Сразу возвращает `outputCh` наружу.

Таким образом, `Transform` не ждёт окончания обработки — он возвращает канал «для будущих результатов», а обработка идёт в фоне.

## Нюансы и принятые решения

### 1. **Generic (параметризация типом `T`)**
- **Почему:** Чтобы `Transform` мог работать с каналами любых типов (`int`, `string`, своей структурой и т.д.) без дублирования кода.
- **Нюанс:** В старых версиях Go без generics пришлось бы писать отдельные функции для каждого типа или использовать `interface{}` (потеря типобезопасности).

### 2. **Направленные каналы (`<-chan T` и `chan<- T`)**
- `<-chan T` — канал, из которого **можно только читать**.
- `<-chan T` в возврате означает, что вызывающий код не может записать данные в этот канал (только читать). Это защита от случайной поломки конвейера.
- Внутри горутины используется запись в `outputCh` (обычный канал), но наружу возвращается канал «только для чтения».

### 3. **Закрытие выходного канала (`defer close(outputCh)`)**
- **Зачем:** Чтобы цикл `range` в `main` завершился, когда все данные обработаны. Без закрытия `range` завис бы навсегда, ожидая новых данных.
- **Нюанс:** Закрывать канал должен **только отправитель**. Здесь отправитель — горутина внутри `Transform`, поэтому она и закрывает.

### 4. **Горутина внутри `Transform`**
- **Почему не делать обработку синхронно?** Если бы `Transform` сама читала и писала без горутины, то она бы заблокировала вызывающий код до полного завершения. С горутиной `Transform` возвращается мгновенно, а обработка идёт параллельно.
- **Нюанс:** Горутина живёт, пока не дочитает входной канал. Если входной канал никогда не закроется, горутина будет висеть вечно (утечка). Поэтому важно, чтобы **отправитель** в `main` закрыл канал (`defer close(channel)`).

### 5. **Порядок и буферизация**
- Каналы в примере **небуферизированные** (ёмкость 0). Отправка блокирует отправителя до тех пор, пока получатель не готов. Это обеспечивает синхронизацию: горутина-писатель и горутина-трансформер работают в такт.
- Если бы канал был буферизирован, могло накопиться несколько значений, но это не меняет логику.

### 6. **Функция-преобразователь вызывается для каждого элемента**
- Она **не должна** сохранять состояние между вызовами (если только это не нужно специально). В примере она чистая (зависит только от входного значения).

### 7. **Потенциальная проблема: медленное преобразование**
- Если `transformFunc` работает долго, то вся цепочка затормозится, потому что следующее значение из входного канала не будет прочитано, пока текущее не обработается. Для параллельной обработки можно было бы запустить несколько воркеров, но здесь простой последовательный вариант.

## Что это за паттерн?

Это аналог функции `map` из функциональных языков программирования, но для потоков данных в Go. Такой подход позволяет строить цепочки обработки:

```go
numbers := getNumbers()          // канал исходных чисел
squares := Transform(numbers, func(x int) int { return x*x })
cubes := Transform(squares, func(x int) int { return x*x })
```

Каждый вызов `Transform` добавляет новый этап обработки, не смешивая логику.

## Итог

- `Transform` — полезный строительный блок для **конвейерной обработки данных**.
- Он отделяет логику преобразования от управления каналами и конкурентностью.
- Принятые решения (generic, направленные каналы, закрытие канала внутри горутины) делают код типобезопасным, надёжным и удобным для переиспользования.
