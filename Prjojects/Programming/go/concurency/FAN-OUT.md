**Область применения**: Шардирование

```go

package main

import (
	"fmt"
	"sync"
)

func SplitChannels[T any](inputChls <-chan T, n int) []<-chan T {
	outputChls := make([]chan T, n)
	for i := 0; i < n; i++ {
		outputChls[i] = make(chan T)
	}

	go func() {
		idx := 0
		for value := range inputChls {
			outputChls[idx] <- value //может быть не блокирующим (select)
			idx = (idx + 1) % n //или вообще выбираем по ключу

		}

		for _, ch := range outputChls {
			close(ch)
		}

	}()

	//can't cast []chan T to []<-chan T
	resultChl := make([]<-chan T, n)
	for i := 0; i < n; i++ {
		resultChl[i] = outputChls[i]
	}

	return resultChl
}

func main() {
	channel := make(chan int)

	go func() {
		defer func() {
			close(channel)
		}()

		for i := 0; i < 100; i += 3 {
			channel <- i
		}
	}()

	channels := SplitChannels(channel, 2)

	var wg sync.WaitGroup
	wg.Add(2)

	go func() {
		defer wg.Done()
		for value := range channels[0] {
			fmt.Println("ch1:", value)
		}
	}()

	go func() {
		defer wg.Done()
		for value := range channels[1] {
			fmt.Println("ch2:", value)
		}
	}()

	wg.Wait()

}
```

Эта программа на Go реализует **разделение потока данных** из одного канала в несколько. Простыми словами: есть один конвейер, с которого сходят детали, и робот-раздатчик раскладывает их по очереди в разные коробки (первую, вторую, потом снова первую и т.д.).  

---

## Что происходит в `SplitChannels`?

1. **Создаём n пустых коробок (каналов)** – в них будем складывать значения.
2. **Запускаем фонового работника (горутину)**, который:
   - Берёт значение из входного канала (`inputChls`)
   - Кладёт его в текущую коробку (`outputChls[idx]`)
   - Переключает указатель на следующую коробку по кругу (`idx = (idx+1) % n`)
   - Когда входной канал закрывается и все значения розданы, **закрывает все выходные каналы** – это сигнал для получателей, что больше ничего не будет.
3. **Возвращает слайс этих каналов**, но помеченных как «только для чтения» (`<-chan T`). Это защита: никто снаружи не сможет случайно записать данные в выходные каналы.

---

## Нюансы и принятые решения

### 1. Дженерики (`[T any]`)
Функция работает с каналами любого типа – хоть `int`, хоть `string`, хоть своей структурой. Универсальность.

### 2. Почему нельзя вернуть `[]chan T` напрямую?
В Go типы `[]chan T` и `[]<-chan T` – **разные**. Нельзя просто так вернуть один вместо другого, даже если внутри лежат одни и те же каналы. Поэтому создаётся новый слайс `resultChl` и в него копируются ссылки на каналы (это безопасно, так как `chan T` можно присвоить переменной типа `<-chan T`).

### 3. Отдельная горутина – неблокирующий запуск
Если бы мы распределяли значения в той же горутине, вызывающий код застыл бы до тех пор, пока не будет прочитан весь входной канал. Так мы отдаём управление и продолжаем работу.

### 4. Round‑robin (по кругу)
`idx = (idx + 1) % n` – классика. Первое значение идёт в канал 0, второе в канал 1, третье снова в 0 и т.д. Это **простейшее шардирование** без ключей или хешей.

### 5. Закрытие выходных каналов
Очень важно: **закрывает тот, кто пишет**. Как только входной канал опустел и закрылся, цикл `for value := range inputChls` завершается, и мы закрываем каждый выходной канал. Это позволяет читателям (в `main`) спокойно выйти из своих циклов `range`.

### 6. Потенциальная блокировка (deadlock)
В текущем коде все выходные каналы **небуферизированные**. Если какой-то из них никто не читает, запись в него навсегда заблокирует раздатчика. В этой программе обе выходных горутины в `main` активно читают, поэтому всё работает.  
Но в общем случае это риск. Решения:
- Сделать каналы буферизированными.
- Использовать `select` с `default` для неблокирующей записи (и потерять данные, если канал переполнен).
- Гарантировать, что каждый выходной канал будет прочитан.

### 7. Комментарий «может быть не блокирующим»
Автор намекает, что запись `outputChls[idx] <- value` **является блокирующей**, но в принципе можно было бы сделать неблокирующий вариант (например, через `select`). В этой реализации – блокирующая.

### 8. Что в `main`?
- Создаётся канал `channel`, в него пишутся числа 0, 3, 6, … 99 (всего 34 числа).
- Запускается `SplitChannels(channel, 2)`.
- Две горутины читают из двух выходных каналов и печатают значения.
- `sync.WaitGroup` нужна, чтобы `main` не завершилась раньше, чем горутины напечатают всё.

### 9. Порядок вывода
Из-за round‑robin:  
- Канал 0 получит 0, 6, 12, …  
- Канал 1 получит 3, 9, 15, …  
Но печать может идти вперемешку, потому что две горутины работают конкурентно. В этом нет ошибки, это особенность параллельного вывода в консоль.

---

## Главная идея
**Разделить поток данных на несколько параллельных подпотоков (шардов), чтобы обрабатывать их независимо.** Это полезно для балансировки нагрузки, например, когда один обработчик не успевает, а два – справляются.
