**Область применения**: объединение нескольких источников данных в одну трубу.

```go

package main

import (
	"fmt"

	"golang.org/x/sync/errgroup"
)

func MergeChannels[T any](channels ...<-chan T) <-chan T {
	outputCh := make(chan T)

	var eg errgroup.Group
	//eg, ctx := errgroup.WithContext(ctx) (альтернатива если нужен контекст)


	for _, channel := range channels {
		//ch := channels (создание локальной копии не нужно начиная с go 1.22, но можно использовать для надежности)
		//*в Go 1.21- в таком цикле каждая channel не была отдельной копией
		eg.Go(func() error {
			for value := range channel {
				outputCh <- value
			}
			return nil
		})

	}

	go func() {
		if err := eg.Wait(); err != nil {
			panic(err)
		}
		close(outputCh)
	}()

	return outputCh
}

func main() {
	channel1 := make(chan int)
	channel2 := make(chan int)
	channel3 := make(chan int)

	go func() {
		defer func() {
			close(channel1)
			close(channel2)
			close(channel3)
		}()

		for i := 0; i < 100; i += 3 {
			channel1 <- i
			channel2 <- i + 1
			channel3 <- i + 2
		}
	}()

	for value := range MergeChannels(channel1, channel2, channel3) {
		fmt.Println(value)
	}

}

```

## Объяснение простыми словами

Представьте, что у вас есть три крана с водой (каналы-источники). Вы хотите получить одну трубу, из которой течёт вода сразу из всех трёх кранов, причём в том порядке, в котором вода поступает. Это и есть **FAN-IN** – объединение нескольких потоков данных в один.

Код делает следующее:
- Создаёт общий выходной канал `outputCh`.
- Для каждого входного канала запускает отдельную «помощницу» (горутину), которая читает все значения из своего канала и отправляет их в общий канал.
- Как только все помощницы закончат (все входные каналы закрыты и опустошены), общий канал закрывается.
- В `main` данные отправляются в три канала, а затем читаются из объединённого канала и печатаются.

## Нюансы и принятые решения

### 1. Почему `errgroup.Group`, а не обычный `sync.WaitGroup`?
- `errgroup` умеет не только ждать завершения горутин, но и собирать ошибки. Здесь горутины всегда возвращают `nil`, но решение оставлено на будущее – легко добавить обработку ошибок (например, если чтение из канала может провалиться).
- Можно было бы использовать `WaitGroup`, но код стал бы чуть длиннее (пришлось бы вызывать `Add`, `Done`, `Wait` вручную).

### 2. Отдельная горутина для закрытия выходного канала
```go
go func() {
    eg.Wait()
    close(outputCh)
}()
```
Закрытие происходит в фоновой горутине, чтобы не блокировать отправку данных. Если бы мы написали `eg.Wait(); close(outputCh)` прямо в теле `MergeChannels`, то функция не вернула бы канал до тех пор, пока все входные каналы не закроются. А это может быть никогда, если пишущая сторона ещё не закончила. Кроме того, `outputCh` нужно вернуть сразу, чтобы вызывающий код мог начать читать из него.

### 3. Захват переменной цикла (актуально для Go < 1.22)
```go
for _, channel := range channels {
    eg.Go(func() error {
        for value := range channel { // channel – "замыкание" на итерационную переменную
```
В старых версиях Go переменная `channel` переиспользовалась на каждой итерации, и все горутины могли получить последний канал из слайса. Поэтому требовалась локальная копия:
```go
ch := channel
eg.Go(func() error {
    for value := range ch { ... }
})
```
Начиная с Go 1.22 переменная цикла создаётся заново для каждой итерации, так что проблема решена. Комментарий в коде предупреждает об этом историческом нюансе.

### 4. Небуферизированный выходной канал
```go
outputCh := make(chan T) // без буфера
```
Это означает, что отправка в `outputCh` блокируется, пока читатель не готов принять значение. Такой подход прост и предсказуем. Если бы канал был буферизированным, могла бы вырасти задержка, но и производительность могла бы стать выше. Выбор в пользу небуферизированного – классический для fan-in.

### 5. Паника при ошибке в `eg.Wait()`
```go
if err := eg.Wait(); err != nil {
    panic(err)
}
```
В учебном примере это допустимо. В реальном коде лучше либо возвращать ошибку через отдельный канал, либо логировать и корректно завершать работу. Здесь ошибка всегда `nil`, поэтому паники не случится.

### 6. Закрытие входных каналов в `main`
```go
defer func() {
    close(channel1)
    close(channel2)
    close(channel3)
}()
```
Это критично! Если входные каналы не закрыть, горутины в `MergeChannels` будут вечно ждать новых данных в `range channel`, и `eg.Wait()` никогда не завершится. Выходной канал останется открытым, а программа зависнет.

### 7. Тип `...<-chan T` и возвращаемый `<-chan T`
- Троеточие (`...`) позволяет передать любое количество каналов: `MergeChannels(ch1, ch2, ch3)`.
- Стрелка слева (`<-chan T`) означает «канал только для чтения». Это безопасно: вызывающий код не сможет случайно отправить данные в выходной канал, а входные каналы гарантированно не будут изменены.

### 8. Порядок данных не сохраняется
Данные из разных входных каналов перемешиваются в порядке готовности. Если вам важен порядок (например, сначала все из канала 1, потом из канала 2), эта схема не подойдёт.

## Итог
Код реализует надёжный и идиоматичный для Go fan-in с использованием `errgroup` для синхронизации. Главные моменты, на которые стоит обратить внимание:
- Закрытие входных каналов обязательно.
- Закрытие выходного канала – в отдельной горутине.
- В старых версиях Go нужно копировать переменную цикла.
- Выходной канал небуферизирован, что даёт естественное управление потоком.
