**Область применения**: создаем копию потока как в сетевом фильтре (двойник-тройник и иже с ними)

```go

package main

import (
	"fmt"
	"sync"
)

func Tee[T any](inputChls <-chan T, n int) []<-chan T {
	outputChls := make([]chan T, n)
	for i := 0; i < n; i++ {
		outputChls[i] = make(chan T)
	}

	go func() {
		for value := range inputChls {
			for i := 0; i < nl i++ {
				outputChls[i] <- value //can be non-blocking
			}
		}

		for _, ch := range outputChls {
			close(ch)
		}

	}()

	//can't cast []chan T to []<-chan T
	resultChl := make([]<-chan T, n)
	for i := 0; i < n; i++ {
		resultChl[i] = outputChls[i]
	}

	return resultChl
}

func main() {
	channel := make(chan int)

	go func() {
		defer close(channel)
		for i := 0; i < 5; i++ {
			channel <- i
		}
	}()
	
	var wg sync.WaitGroup
	wg.Add(2)

	channels := Tee(channel, 2)

	

	go func() {
		defer wg.Done()
		for value := range channels[0] {
			fmt.Println("ch1:", value)
		}
	}()

	go func() {
		defer wg.Done()
		for value := range channels[1] {
			fmt.Println("ch2:", value)
		}
	}()

	wg.Wait()

}
```

Этот код реализует **разветвитель потока данных** (аналог тройника для трубы): все значения, которые приходят в один канал, копируются и отправляются в несколько других каналов.

## Что делает функция `Tee`

1. Создаёт `n` новых выходных каналов.
2. Запускает фоновую горутину, которая:
   - читает значения из входного канала (`inputChls`)
   - каждое значение отправляет **во все** выходные каналы
   - когда входной канал закрывается, закрывает все выходные
3. Возвращает эти `n` каналов (но только для чтения, чтобы никто случайно не закрыл их раньше времени).

В `main` канал с числами 0…4 раздваивается, два потребителя параллельно читают свои копии и печатают их.

## Нюансы и принятые решения

### 1. Почему один медленный потребитель тормозит всех?
Отправка в каждый выходной канал — **блокирующая** (каналы без буфера). Если один из потребителей не читает данные, горутина `Tee` застрянет на отправке в его канал. Из‑за этого не будут отправлены значения и в остальные каналы, хотя те готовы читать.  
🔧 В коде есть пометка *«can be non‑blocking»* — можно переделать через `select` с `default`, чтобы пропустить отправку, если канал занят, но тогда данные теряются.

### 2. Зачем два среза с каналами: `outputChls` и `resultChl`?
- Внутри функции нам нужны **двунаправленные** каналы (`chan T`), чтобы писать в них.
- Наружу мы отдаём **каналы только для чтения** (`<-chan T`), чтобы пользователь случайно не закрыл или не записал в них сам.
- В Go срез `[]chan T` и срез `[]<-chan T` — разные типы, их нельзя просто так вернуть. Приходится создавать новый срез и копировать элементы.

### 3. А что если `n = 0`?
Функция создаст пустой срез, горутина будет читать входной канал впустую, затем закроет нуль каналов. Это не ошибка, но чаще всего `n >= 1`. Можно было бы добавить проверку и паниковать, но разработчик оставил поведение на усмотрение вызывающего.

### 4. Кто закрывает каналы и когда?
Все выходные каналы закрывает сама горутина `Tee` **после того, как входной канал закрыт и все значения из него прочитаны**. Это гарантирует, что потребители смогут выйти из цикла `range`.  
⚠️ Если входной канал никогда не закроется, горутина `Tee` будет висеть вечно → утечка памяти.

### 5. Почему в `main` используется `sync.WaitGroup`?
Чтобы дождаться, пока обе горутины‑потребителя напечатают все значения. Без `WaitGroup` программа могла бы завершиться раньше, потому что главная горутина не ждёт остальные.

### 6. Порядок доставки
Значения копируются в том же порядке, в котором приходят. Каждый потребитель получает свою копию потока, но из‑за конкурентного чтения моменты печати могут чередоваться.

### 7. Почему каналы не буферизованы?
Это упрощает логику и гарантирует синхронность: отправка не произойдёт, пока получатель не готов. Если нужна бóльшая пропускная способность, буфер можно добавить, изменив `make(chan T)` на `make(chan T, size)`.

## Итог простыми словами

Код делает из одного «водопровода» несколько параллельных, копируя каждую порцию данных во все отводы. При этом если хотя бы один отвод забит, вся система встаёт. Такое решение типично для паттерна «tee» (тройник) в Go.
